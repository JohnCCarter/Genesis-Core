# RI router replay — first concept-lane case

Date: 2026-04-23
Mode: `RESEARCH`
Branch: `feature/ri-role-map-implementation-2026-03-24`

## Why this is the right first concept case

If the three-lane model is going to be useful, the first real test should be a candidate that is:

- genuinely tempting to over-integrate too early,
- already surrounded by partial research evidence,
- still small enough to falsify without opening runtime structure.

The RI router replay candidate fits that profile.

The repository already has evidence that:

- a **full** RI role-map baseline is still too expensive and not yet honest on the current evidence surface,
- a **bounded** deterministic baseline is admissible on narrower surfaces,
- a docs-only RI-local router replay can be defined below runtime integration.

That makes this a strong first concept-lane case: it tests whether the idea can live as a cheaper non-runtime shape before anyone tries to turn it into family/runtime authority.

## Concept framing

- **Lane:** `Concept`
- **Objective:** test whether RI role-map thinking should first be expressed as a deterministic RI-local router replay plus bounded trace/evaluation surfaces, rather than as new runtime-family structure.
- **Candidate:** `RI router replay as non-runtime role-map container`
- **Why this is the cheapest admissible lane now:** the question is still about idea shape, falsification, and surface cost — not runtime semantics, not defaults, and not authority.

## Hypothesis / idea

Before Genesis creates any new durable RI runtime/family surface, RI role-map ideas should first survive as a cheaper pair of non-runtime research shapes:

1. an RI-local deterministic router replay that emits routing / veto / dwell / no-trade traces, and
2. a bounded deterministic baseline that evaluates only the admissible pre-entry and trade-level surfaces already supported by current evidence.

If those two cheaper forms fail to produce stable, honest, and reproducible signal, then creating runtime structure earlier would be premature.

## Why this might be better

This concept is attractive because it keeps several expensive mistakes out of the runtime lane:

- it avoids introducing a new strategy family just to give the idea a container,
- it avoids turning partial RI evidence into premature authority language,
- it lets the repository test policy-selection thinking as a replay/trace problem first,
- it makes `2024` vs `2025` contradiction visible before runtime semantics ossify,
- it keeps failure cheap if the role-map idea turns out to be thin, unstable, or mostly unsupported.

## What would falsify the concept

The concept should be considered falsified, or at least not ready for evidence promotion, if any of the following turn out to be true:

1. the router replay cannot be defined deterministically from RI-local decision-time state alone,
2. the replay requires cross-family logic, hidden score composition, or runtime-path assumptions to look coherent,
3. the bounded baseline remains mostly unsupported, weakly anchored, or degenerate once explicit coverage reporting is applied,
4. `2024` and `2025` tell materially contradictory stories that collapse the usefulness of the proposed surfaces,
5. the idea only appears viable when expressed as runtime/family semantics instead of as replay/bounded evidence surfaces.

## Cheapest admissible surfaces

This concept should stay on the following surfaces first:

- `docs/analysis/**`
- existing `results/research/**` artifacts
- replay/trace design and manifest planning
- bounded research scripts under `tmp/**`
- read-only comparisons against frozen/tracked artifacts

It should stay out of:

- `src/**`
- `tests/**`
- `config/**`
- champion/default authority
- runtime router integration
- paper/live execution surfaces

## Why this is still concept lane rather than research-evidence lane

Even though some admissibility and replay-planning work already exists, the core question here is still conceptual:

- **what is the cheapest honest container for RI role-map ideas?**

That question is not yet about proving final improvement.
It is still about deciding whether the idea should live first as:

- replay profile,
- bounded baseline,
- role-map candidate,

instead of as runtime/family structure.

So this slice should stay concept-only until the repository explicitly opens a bounded evidence packet for one of those cheaper forms.

## Recommended next bounded evidence step if the concept survives

If the concept survives review, the next smallest research-evidence slice should be:

- **one bounded RI router replay evidence slice**

That slice should do only this:

1. freeze one replay input manifest,
2. replay one deterministic RI-local router on eligible RI decision points only,
3. emit canonical routing / veto / dwell / no-trade traces,
4. report `2024` and `2025` separately,
5. stop if routing quality depends on forbidden fields or hidden runtime assumptions.

Only after that should the repository decide whether a further bounded deterministic baseline slice is worth opening on the admitted trade-level surfaces.

## Explicit non-goals

This concept case does **not** propose:

- a new strategy family,
- runtime authority,
- champion/default changes,
- production readiness,
- promotion framing,
- ML comparison,
- cross-family routing.

## Bottom line

The first real concept-lane case should be:

- **RI router replay as a non-runtime container for RI role-map ideas**

because it is:

- already partially anchored by existing docs,
- naturally falsifiable,
- cheaper than runtime structure,
- and exactly the kind of idea that the lane model is supposed to keep out of expensive family/runtime form until it has earned that cost.
