# Genesis Topology / Lifecycle / Authority Map

> This document is a derivative, citation-bound orientation map. It does not create authority, does not override cited sources, and any unsupported area remains `UNRESOLVED`.

## Purpose

This map answers one bounded repository question only:

- how the current Genesis-Core topological zones and key control surfaces relate as active working areas, derivative/support surfaces, output-hygiene zones, or retained historical/context surfaces
- what lifecycle posture and authority posture each directly supported zone carries
- what advisory `next-step bias` and process-speed band is safest when operating near that zone

It may be used for navigation and repo-orientation only.
It may **not** be used as an independent SSOT, authority source, approval surface, or exhaustive file-by-file classifier.
Inclusion of a zone here does not by itself grant current operational authority to the files, conclusions, or historical materials that happen to live inside that zone.

## Scope boundary

This map covers only the following directly supported surfaces:

- cross-zone authority and control anchors
- top-level working zones
- top-level support, integration, and classification zones
- output/hygiene zones
- historical planning/context zones with direct current-status support

This map does **not**:

- classify every file or every nested subfolder
- change governance precedence, mode resolution, or higher-order authority
- authorize runtime changes, promotion, readiness, or config interpretation
- prove lineage, authority, or dependency from folder placement, chronology, or prominence alone

If a zone or surface is not directly supported by current cited text or explicit observed tree structure paired with a current-use source, the honest state here is `UNRESOLVED`.

## Reading rule

Resolve current authority first from the explicit user request and the higher-order governance stack.
Use this map only after that step, as a derivative repo-zone orientation aid.

If a cited source conflicts with this map, the cited source controls.
Topology, lifecycle, authority posture, risk posture, and `next-step bias` here are zone-level orientation only.
Files and materials inside a mapped zone remain non-authoritative by default unless current authority is established elsewhere by explicit listing, active-authority citation, or promotion through a current governance packet.

`Next-step bias` is an advisory working cue inferred from cited sources; it is **not** an approval state, permission grant, or governance override.
The `GREEN / YELLOW / RED` speed lens below is derivative shorthand only; it does **not** replace the higher-order governance stack, mode resolution, or cited source controls.

## Column guide

- **Observed current role** — what the zone or surface currently appears to do in the checked-in tree plus cited current docs
- **Lifecycle posture** — whether the zone is primarily active, mixed, retained historical, output/hygiene, local support, or unresolved
- **Authority posture** — whether the zone acts as active authority in a bounded domain, a derivative/non-authorizing aid, an evidence/output surface, or remains unresolved
- **Misread / change risk** — how costly misunderstanding or casual editing is likely to be
- **Advisory next-step bias** — the safest default operating cue for future work near that zone

## Practical speed / governance lens

This map may also be read through a simple advisory speed lens tied to behavior-change risk and proximity to runtime, promotion, config-authority, and data-semantics surfaces.

- `GREEN` — fast path for low behavior-change-risk documentation work such as docs-only edits, status notes, maps, and routing docs.
- `YELLOW` — medium path with reference checks for archive/move candidates, config classification, and scripts lifecycle work where lineage or current-use drift is plausible even when direct runtime behavior change is not the goal.
- `RED` — strict path for runtime, optimizer, strategy-family, data-semantics, or promotion-adjacent work where behavior, authority interpretation, or readiness claims can be affected.

Advisory reading rule:

- the lower the risk of behavior change, the faster the admissible process
- the closer the slice is to runtime, promotion, config authority, or data semantics, the harder the governance should be
- if a slice straddles bands or the cited support is mixed, bias upward rather than downward

## 6. Practical use sequence

Use this map in the following order only:

1. resolve current authority from the explicit user request and the higher-order governance stack
2. identify the touched zone or control surface
3. read that row here for lifecycle posture, authority posture, risk posture, and advisory `next-step bias`
4. use the `GREEN / YELLOW / RED` speed lens above to size the process conservatively for the slice
5. if the zone or claim is unsupported here, treat it as `UNRESOLVED` and open a later bounded slice rather than inferring
6. drop into the cited zone-local source docs before making a claim or change
