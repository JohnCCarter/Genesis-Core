# Documentation Provenance & Lineage Map

> This document is a derivative, citation-bound orientation map. It does not create authority, does not classify the entire repository, and leaves uncited or deep corpora unresolved unless directly classified by a cited source.

## Purpose

This map answers one bounded documentation question only:

- how the current Genesis-Core documentation surfaces relate to one another as active governance references, non-authorizing routing guides, derivative knowledge-layer surfaces, or retained historical/provenance surfaces

It may be used for navigation and provenance/origin context.
It may **not** be used as an independent SSOT, authority source, or proof that one document silently governs another.
If a cited source conflicts with this map, the cited source controls.
Inclusion, omission, chronology, filename prominence, folder placement, or index placement do not by themselves imply lineage, dependency, approval, or current applicability.

## Scope boundary

This map covers only the following directly cited documentation surfaces:

- top-level governance and workflow references
- top-level documentation routing guides
- current derivative knowledge-layer documents
- high-visibility historical/provenance surfaces with explicit current-use notes

This map does **not**:

- classify every packet under `docs/decisions/**`
- classify every note under `docs/analysis/**`
- prove lineage, authority, or dependency from chronology alone
- prove lineage, authority, or dependency from filename prominence, folder placement, or index inclusion alone
- create runtime, readiness, promotion, or governance authority

Deep or uncited corpora remain `UNRESOLVED` in this map unless a later bounded slice classifies them directly.

## Reading rule

Read authority from the cited source artifacts themselves.
Use this map only as a derivative orientation aid after current authority has already been resolved from the higher-order governance stack.

If a surface is not directly supported by cited current text, the honest state here is `UNRESOLVED` rather than guessed lineage.

## 6. Practical use sequence

Use this map in the following order only:

1. resolve current task authority from the explicit user request and the active governance stack
2. use `docs/CURRENT_AUTHORITY_INDEX.md` to see which current documentation surfaces are active, non-authorizing, or dormant in their cited role
3. use this map to understand how those current surfaces relate to nearby routing guides and historical/provenance documents
4. descend into zone guides (`docs/README.md`, `docs/governance/README.md`, `docs/analysis/README.md`, `docs/decisions/README.md`) only for navigation and placement
5. read historical/provenance documents only within their own current-use boundary
